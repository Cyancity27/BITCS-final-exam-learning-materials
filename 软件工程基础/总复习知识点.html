1 - 基本软件过程的3个模型 Software Process Models

中文版 P17

英文版 P29 （PDF47)

瀑布模型 Waterfall

也即软件的生命周期模型。该模型将基本的过程活动，包括描述、开发、有效性验证和进化，看成是一些界限分明的独立的过程阶段。

五个阶段：需求分析和定义、系统和软件设计、实现和单元测试、集成和系统测试、运行和维护。

迭代开发模型 RUP

zyc觉得（这个似乎不是基本模型，是一个比较新的模型）

增量式开发 Incremental

增量式开发的思想是先开发出一个初始的实现，给用户使用并听取用户的使用意见和建议，通过对多个版本的不断修改直到产生一个充分的系统。

描述、开发和有效性验证等活动不是分离的而是交织在一起。同时让这些活动之间都能得到快速的反馈信息传递。系统的开发是建立一系列的版本（增量)，每个版本添加部分功能到先前的版本中。

面向复用的软件工程 Reuse oriented

该方法是基于已存在的大量可复用的组件。系统开发过程着重于集成这些组件到新系统中，而非从头开发。

需求描述、组件分析、需求修改（需求对应已有组件做出妥协）、使用复用的系统设计、开发和集成、系统有效性验证

2 - 软件工程和其它工程的差异

中文版 P3

英文版 P5 （PDF23）

和计算机科学的差异

计算机科学侧重理论和基础，而软件工程侧重软件开发和交付的实际活动。

和系统工程的差异

系统工程侧重于计算机系统的开发的所有方面，包括硬件、软件和过程工程，软件工程只是这个总体过程中的一部分。涉及开发系统中的软件基础设施、控制、应用程序和数据库。

3 - 风险评估 Risk Analysis

中文版 P379

英文版 P599 （PDF 616）

  1. 对风险出现的可能性（probability of the risk）进行评估，可能的结果有3+2种：

    风险出现的可能性非常低（<10%)、低( 10% ~25%)、中等（25% ~50%)、高（ 50% ~75%）或非常高（>75%)。

  2. 对风险的严重性（effects of the risk）进行评估，可能的结果有4种：

    灾难性的（严重威胁项目的存活）、严重的（可能引起很大的延迟）、可以容忍的（可能引起小的延迟）和可以忽略的。

（very low、low、moderate、high、very high）
（catastrophic、serious、tolerable、insignificant）

6 - 软件产品的属性 Product Characteristics

英文版 P8 （PDF 25）

中文版 P4

（优良软件的特点：可维护、可靠、可用）

  • Maintainability 可维护性

    能够不断进化，满足需求的变化。是软件产品的最根本特性。

  • Dependability and security 可依赖性和安全性

    在系统失败的情况下不会导致物理性损害和经济损失，有恶意的人员不能访问或破坏系统。

  • Efficiency 有效性

    响应时间、处理时间和内存利用率等方面。不浪费资源。

  • Acceptability 可用性

    简单易用，① 容易被用户接受，且 ② 与其它系统兼容。

9 - 软件开发工具(CASE tools)的功能 >

英文版 P37 （PDF 54）

中文版 P22

软件开发工具（有时也叫计算机辅助软件工程或CASE工具）是用来支持软件工程过程活动的程序，这些工具包括如：设计编辑器、数据字典、编译器、调试器、系统生成工具等。
软件工具通过自动化某些过程活动和提供正被开发的软件的信息来支持软件过程。可以自动化的活动的例子包括：

  • 作为需求描述和软件设计中的一部分的图形系统模型的开发。
  • 从这些图形模型生成代码。
  • 通过由用户交互创建的图形界面描述来产生用户界面。
  • 通过提供执行的程序的信息进行程序调试。
  • 程序的自动转换，将使用旧语言版本的程序转换成一个新版本语言的程序。

多种工具可能被整合在一个被称为交互式软件环境 (叶语霄改：交互式开发环境) 的框架中，也称为IDE。这就为不同的工具提供了一个共同的可用的设施集合，使之更容易在集成的方式下通信和运行。人们广泛使用ICLIPSE IDE，它嵌入了很多不同类型的软件工具。

programs that are used to support software engineering process activities
比如：
Graphical editors for system model development; （模型的图的建立与编辑）
Data dictionary to manage design entities; （数据管理）
Graphical UI builder for user interface construction; （GUI建立）
Debuggers to support program fault finding; （程序调试功能）
Automated translators to generate new versions of a program.（版本生成）

10 - 组件接口

英文版 P457 （PDF 474)

中文版 P

  • 组件
    1. 组件是一种软件元素，与某个组件模型要求相一致，按照组成标准无需修改即可独立进行部署和组合。（基于标准）
    2. 组件是具有合同定义的接口和显式的上下文依赖，可独立进行部署的并服从于第三方的组成的软件单元。（基于关键特征）

可复用组件的2个关键特征：

1．组件是独立可执行的实体，这是由它的接口定义的。你不必知道组件的源代码的任何信息来使用它。它可能作为一种外部服务来引用，也可能直接包含在一个程序中
2．组件所提供的服务可以通过其接口得到，而且所有的交互都是通过接口实现的。组件接口表现为参数化的过程，其内部状态是不会暴露出来的。

  • 组件接口
    • 提供接口 Provides Interface：提供接口定义了组件所提供的服务。
    • 需要接口 Requires Interface：需要接口如果一个组件要进行正确的操作，指定系统其他组件必须提供哪些服务。

11 - Reengineering 软件再工程 >

  • 英文 P248
  • 中文 P158

对于很多系统，特别是遗留的老系统，它们很难理解和进行变更。这些程序可能最初牺牲了一些可理解性来换取在性能上或空间利用上的改善，另外，随着时间的推移最初的程序结构经过一系列的变更后已被破坏了。
为了使得遗留系统的维护问题变得更简单，你可以再工程这些系统以增强它们的结构性和可理解性。

再工程包括

  1. 对系统重新建立文档、
  2. 重构系统体系结构、
  3. 用一种更先进的程序设计语言转换系统、
  4. 修改和更新系统的数据结构和系统的数据取值。

一般来讲，软件的功能不会改变，也应当避免对系统体系架构的大的改动。

（叶语霄补）再工程过程中的活动如下：

  1. 源代码转换。 使用转换工具，将程序从旧的程序设计语言转换到相同语言的较新版本或另一种语言。
  2. 反向工程 。对代码进行分析，抽取其组织结构和功能。该过程通常是全自动完成的。
  3. 程序结构改善。 对程序的控制结构进行分析和修改，使它更容易读和理解。
  4. 程序模块化。 程序的相关部分被收集在一起，在一定程度上消除冗余。在某些情况下，这个阶段可能包括体系结构的重构。
  5. 数据再工程。 改变程序处理的数据以反映程序变更。

12 - Project Management 项目管理

英文版：P594

中文版：P376

项目管理包括

  • 风险管理
  • 人员管理

风险管理 Risk Management

风险管理要求项目管理者：能预见可能影响正开发的软件的项目进度或产品质量的风险，并采取行动避免这些风险，是项目管理者的一项重要任务。

  1. 项目风险(Project Risk)︰是影响项目进度或项目资源的风险。

    比如失去一位经验丰富的设计师。寻找一位经验和技能满足要求的替代者可能需要很长一段时间，而这带来的直接后果就是软件的设计需要更长的时间才能完成。

  2. 产品风险(Product Risk)：是影响开发中软件的质量或性能的风险。

    例如，购买的组件不能达到预期目标。这可能会影响到整个系统的性能，导致整个系统运行速度下降。

  3. 业务风险(Business Risk)：是影响软件开发机构或软件产品购买机构的风险。

    例如，竞争对手推出了新的产品。竞争产品的推出可能预示着先前做出的关于现有软件产品的销售情况可能过于乐观。

当然，这种分类方式会有重叠。如果富有经验的程序设计师离职，由此带来的风险可能是项目风险。即使能立刻找到替代人员，进度也将受到影响。新来的项目成员不可避免要花一些时间来熟悉这个项目，并不能立刻高效地投入工作。因此，系统的交付可能延期。同时这也会是产品风险，因为继任者可能不如前者有经验，因而可能犯错误。最后，它也会是业务风险，因为有经验的程序员是取得新的合同的关键。

人员管理 Managing People

  1. 一致性：对项目组的每个人应该同等对待。

    虽然没有人会期望每个人的报酬都相同，但是却能感觉自己对机构的贡献遭到低估。

  2. 尊重︰不同的人有着不同的技能，管理者应尊重这些差异。

    团队的所有成员都应有机会做出贡献。当然在某些情况下你会发现某个人根本不适合在这个团队，所以也不能够继续下去，但重要的是不要匆忙得出这样的结论。

  3. 包容：当人们觉得其他人能够倾听并能采纳自己的建议时，他们能更有效地工作。

    很重要的一点是创造一种工作环境，在这种工作环境下，所有的见解，甚至是最底层员工的意见都会得到考虑。

  4. 诚实︰作为一个管理者，必须对项目组中好的情况和不好的情况保持诚实的态度。

    应当诚实地对待自己的技术知识水平，并且在必要的时候乐于服从更博学的员工。如果掩盖存在的问题和自己的无知，最终将被发现并且失去团队的尊敬。

工作动力：

  1. 满足员工的社会需求：给员工提供与同事交往的时间和场所。

  2. 满足员工的受尊重需要：让员工们感觉到他们在开发机构中很受尊重。成绩认同、薪资反馈等。

  3. 满足员工的自我实现需要：让员工对自己的工作负起责任，分配给他们比较难的（但不是不能完成的）任务，并给他们提供培训计划以提高他们的技能。

对项目进行衡量的标准：
里程碑（deliverables）：过程活动的终点（往往是各种report, design, requirement）
可交付文档（milestone）：交付给顾客的成果

13 - CORBA Standard

英文版：P482

中文版：P303

PDF(pp12v2)

CORBA（Common Object Request Broker Architecture公共对象请求代理体系结构）是一个闻名的对中间件系统的描述，它的目标是建立一个开放标准允许开发中间件支持分布的组件通信和执行，也提供一系列被这些组件使用的标准服务。
已经有了CORBA的多个实现，但是系统从未达到过临界规模。用户倾向于使用专用的系统或是投向面向服务的体系结构。

CORBA 标准：
①An object model for application objects：
A CORBA object is an encapsulation of state with a well-defined, language-neutral interface defined in an IDL (interface definition language).
②An object request broker that manages requests for object services.
③A set of general object services of use to many distributed applications.
④A set of common components built on top of these services.

14 - 三类 people management （叶语霄添加）

职业人士可分为3种类型：

  1. 面向任务型（Task-oriented）

    这类人的动力来自于他们所从事的工作。软件开发智力上的挑战激发了他们的工作热情。

  2. 面向自我型（Self-oriented）

    这类人的动力主要来自于个人成功和得到认可。他们更乐于把软件开发视为达到自己目标的手段。

  3. 面向交互型（Interaction-oriented）

    这类人的动力来自于同事们的存在和协作。

第三类人员喜欢小组作业，而前两类人员喜欢独自工作。女人比男人更容易称为第三类人员，她们常是更有效的交流者。

每个个体的工作动力由各种因素组成，但在任一时刻总是只有一种动力处于支配地位，并且可能会发生改变。

15 - Software Reuse （软件复用）的好处

中文版 P271

  • 增加可靠性

    在运行着的系统中，已经投入使用的组件可能已经经过各种环境的测试，故已经投入的组件比新组建的可靠性高。

  • 降低过程风险

    如果组件已经存在，复用组件的成本比新开发组件的成本不确定性减少。特别是大规模组件（子系统复用）的复用，极大降低了项目管理中项目成本估计的不确定性。

  • 专家的有效使用

    不用开发者（专家）重复工作，而是通过开发可复用的组件，在其中封装他们的知识。

  • 标准一致

    用某些标准（如用户界面标准）来规范化实现一组标准组件，增强可靠性。因为用户面对相同的操作界面，出现错误的可能性降低。

  • 加速开发

1 - 功能需求与非功能需求（叶语霄添加）

中文版P52
英文版P84（PDF 101）

  • 功能需求

    包括对系统应该提供的服务、如何对特殊输入作出反应、系统在特定条件下的行为的描述。某些情况下功能需求还需声明系统不应该做什么。

①系统应该提供的服务语句，系统应该如何响应特定的输入，以及系统在特定情况下应该如何行为。
②描述功能或系统服务。
③取决于类型的软件,将用户和系统的类型所使用的软件。
④功能用户需求可能是高层陈述了系统应该做什么,但功能系统需求应该详细描述系统服务。

  • 非功能需求

    不直接关系到系统向用户提供的具体服务的一类需求。它们与系统的总体特性相关，如可靠性、响应时间和存储空间占用等。

    是对系统提供的服务或功能给出的约束，包括时间约束、开发过程的约束和所受到的标准的约束。非功能需求经常适用于整个系统，而不是个别的系统功能或服务。

①对系统提供的服务或功能的限制，例如时间限制、开发过程的限制、标准等。
②这些定义了系统属性和约束，例如可靠性、响应时间和存储要求。约束是I/O设备能力、系统表示等。
③流程需求也可能指定要求一个特定情况下系统,编程语言和开发方法。
④非功能性需求比功能需求可能更重要。如果不满足这些条件，系统就没有用处

5 - 分布式组件体系结构（叶语霄添加）

中文版P313
英文版P495（PDF 512）

与Client/Server结构不同的是，不生硬地区分客户机和服务器。每个实体都是一个组件，为其他组件提供服务，也接收来自其他组件的服务。综上，是一种比Client/Server更通用的结构，但自然也更加复杂。（本段内容来自不老师ppt。pp12，slide26）

分布式组件系统依赖于中间件来管理组件间的交互。每个组件给出它所提供的服务的接口，其他组件通过中间件来调用这些服务。

分布式组件体系结构的好处：

  • 允许设计者延迟决断在哪个逻辑层提供服务。没必要实现决定一个服务是属于数据管理层的一部分还是应用层的一部分。
  • 是非常开放的体系结构，系统很容易添加新的服务。
  • 灵活，可扩展。（flexible and scalable）
  • 可以通过组件在网络上的移动，达到系统的动态配置（reconfigure the system dynamically）。例如将提供服务的组件移动请求服务的计算机上，改善系统性能。

在分布式系统中，不同的组件可能用不同语言实现，数据模型、信息表示法、通信协议可能不一样…因此需要某种软件来管理这些部分，确保它们能通信和交换数据。

  2. 提供公共服务，提供很多组件都需要的公共服务的复用实现，例如安全服务（身份验证）、通知服务等。

（补充）
①在逻辑通信层，中间件允许不同计算机上的对象交换数据和控制信息
②在组件级，中间件为开发兼容的组件提供了基础。
③分布式对象系统需要中间件来处理对象通信以及添加和删除系统对象。

3 - 软件复用的优点与问题（周祐超添加）

中文书P271 （PDF285）表格里都有解释

3.1优势：

  • 增加可靠性
    已经运行的系统中的组件往往比新开发一个组件的可靠性要高，因为它们已经得到了各种环境的使用和测试；
  • 降低过程风险
    复用组件的成本不确定性比新开发低，这在现目管理中是一项重要因素；
  • 专家工作的高效使用
    避免专家总是在做重复的工作，只需要封装他们的知识就行；
  • 标准的一致化
    比如GUI界面的菜单栏组件等等，可以实现为一组标准组件，使得用户在使用上更遍历
  • 开发的加速
    毋庸置疑，加快了系统产品的形成，无论是开发还是有效性验证的时间都缩短了；

3.2 问题

  • 增加维护成本
    当组件的源代码不可得的时候，如果系统维护成本增加，那么系统复用部分的不相容性必然再增加
  • 缺乏工具支持
    CASE工具集没有对复用的支持，而将这些工具与组件库系统集成是困难甚至是不可能的。
  • 人员因素
    由于有的程序员更喜欢按照自己的思路重新写一个组件，并且觉得自己有能力去改善这个复用的组件
  • 需要创建并维护一个组件库
    填充组件库，使得组件可用，并保证软件人员能够使用组件库非常耗费财力
  • 寻找并理解可重用组件
    组件本身也需要被寻找，理解，并做一些适当的调整才能使得组件完全适应当前的环境

4.4 设计模型

表示系统中包含的对象，并且描述了实体间的关联与关系。我们需要对模型抽象，避免琐碎的细节，并恰到好处的提供知识便于程序员做出、实现决策。

使用UML帮助设计时，通常需要如下两类设计模型：
1.结构模型：通过系统对象类之间的关系描述静态结构，比如继承、组合、使用/被使用关系

4.5 接口设计

主要指的是定义关于一个或一组对象的详细接口信息，也就是定义对象或对象组所提供的服务标记、语义。
注意，不能含有数据表示的细节，因为接口描述中没有属性的定义，但需要包含能够访问和更新这些数据的操作。并且接口使得其它对象能方便的进行使用与交互，对于数据具体表示是隐藏的。

测试是非常昂贵和费时的。测试案例设计的一个系统化的方法就是对系统或组件识别所有输入和输出的等价划分，被测试的程序应该以同样的方式处理一个输入等价划分中的所有数据。

一旦找出了划分集合，我们就可以从每一个这些划分中选择测试案例。一个选择测试案例的好的经验法则是：在划分的边界上和靠近划分中间点的地方选择测试案例。

  • 划分的中心点是典型值。设计者和程序员在开发系统时倾向于考虑输入典型值。
  • 边界值通常是非典型值（例如零和其他非负数的行为是不同的）。有时被开发者忽视掉，程序经常在运行这些非典型值时失败。

当使用程序描述或用户文档来识别划分，这被称为白盒测试（？）。白盒测试对黑盒测试是一个很好的补充，我们可以看到程序代码来发现可能的测试;当使用系统描述来确定等价划分，这被称为黑盒测试。这时我们不必了解系统是如何工作的。

7 - 紧急变更（emergency changes）

英文版 P239 （PDF 256）
中文版 P152
有下面几个原因使得紧急变更会发生：

  1. 有严重的系统缺陷发生，那么必须修补而使正常的操作继续。
  2. 系统操作环境的变更中有不期望的情况发生，就会破坏正常的操作。
  3. 系统上运行的业务有意料之外的改变发生，比如有新的竞争对手出现或有新的法律生效并影响到了系统。

在这些情况下，就需要做出快速的变更，也意味着你不能遵循正常的变更分析过程。不是按正常的过程去修改需求和设计，而是对程序做紧急的修补来解决突发问题（如下图）。

8 - 独立路径（Independent Path）（叶语霄添加）

英文版P216（PDF 233）
中文版P138

路径测试是一种测试策略，其目标是要执行组件或程序中的每一条执行路径。如果每一条独立路径都执行过，那么组件中的每条语句就肯定至少执行过一遍。此外，所有条件语句的真值状态和假值状态都经过了测试。在面向对象的开发过程中，路径测试在测试对象中的方法时常会用到。

9 - 客户机/服务器结构（Client/Server Architecture）（叶语霄添加）

英文版 P488 (PDF 505)
中文版 P308

C/S结构 （对应教材18.2节）

在client-server模型中，应用被建模为由服务器所提供的一系列服务。

客户机需要知道服务器的存在，而不需要知道其他客户机的存在；服务器也不需要知道客户机的存在。

单个处理器上会运行多个客户机进程。下图说明了12个逻辑的客户机进程运行在6台计算机上的情况：

C/S系统依赖于清晰地分离了信息表示和信息计算（presentation & computation）。我们让信息表示和信息计算分布在不同的逻辑分层上，层与层之间有明确的接口。这种设计允许每一层分布在不同的物理计算机上。将应用分解为如下四层：

  • 表示层（presentation layer）。呈现信息给用户，接收用户的交互。
  • 应用处理层（Application processing layer）。应用逻辑的底层实现。
  • 数据管理层（Data management layer）。管理传给客户机和来自客户机的数据。
  • 数据库层（Database layer）。存储数据。

两层C/S结构 （对应教材18.3.2节）

系统实现为一个独立的逻辑服务器，加上不确定数目的客户机。可分为以下两种形式：

  • 瘦客户机模型（Thin-client model）。表示层在客户机端实现，其他层（应用处理层、数据管理层、数据库层）在服务器上实现。

    缺点是将繁重的计算压力都放在了服务器端，导致C、S之间的网络上有很大的流量。

  • 胖客户机模型（Fat-client model）。表示层+部分或所有应用处理层在客户机端实现。

例子是银行的ATM系统。服务器运行客户账户的数据库，ATM（客户机）上的软件执行很多与用户相关的处理，如提供现金等交易。缺点是系统管理复杂。软件的新版本要在所有客户机上进行更新。

多层C/S结构 （对应教材18.3.3节）

优点：便于管理，不同的逻辑层在不同的处理机上执行；可扩展性更好，当客户数量增加时，增加服务器数量相对容易。

例子是互联网银行系统。这个系统中有3级：大型计算机提供数据库服务；web服务器提供数据管理（管理账户）和应用处理服务（现金转账、生成工资单等）；用户计算机和浏览器构成客户机（表示层）。
